# Design Prompt: เครื่องมือวิเคราะห์เครือข่ายความสัมพันธ์จัดซื้อจัดจ้างภาครัฐ (network.html)

> เอกสารนี้สกัดจากโค้ดจริงของ network.html เพื่อใช้เป็น prompt ตั้งต้นสำหรับ AI หรือทีมออกแบบที่ต้องการสร้าง dashboard ลักษณะเดียวกัน หรือทำ UI audit / redesign

---

## 1. โจทย์และบริบท (Brief)

สร้าง **single-page interactive dashboard** สำหรับนักตรวจสอบภาครัฐไทย ใช้วิเคราะห์ความเสี่ยงในข้อมูลจัดซื้อจัดจ้าง (เครือข่ายหน่วยงาน↔ผู้รับจ้าง, การแบ่งซื้อแบ่งจ้าง, การใช้วิธีเฉพาะเจาะจง, ความผิดปกติด้านราคา) ประมวลผลทั้งหมดฝั่ง client (ไม่มี backend, ไม่มีข้อมูลออกนอกเครื่อง) รันเป็นไฟล์ HTML เดี่ยว โฮสต์บน GitHub Pages ได้

**ผู้ใช้:** ผู้เชี่ยวชาญ (นักตรวจสอบ) ไม่ใช่ผู้ใช้ทั่วไป — ยอมรับความหนาแน่นของข้อมูลสูงได้ แต่ต้องการความแม่นยำ ความน่าเชื่อถือ และไม่ต้องการให้ตัวเลข "ดูฟันธง" เกินจริง

**หลักการที่ต้องสื่อผ่าน UI ตลอดทั้งระบบ (non-negotiable):** ทุกตัวเลข/ธงความเสี่ยงคือ *สัญญาณคัดกรอง* ไม่ใช่ข้อสรุปว่าทุจริต — โทนสี คำอธิบาย และ disclaimer ต้องเสริมข้อความนี้ ไม่ใช่แค่พูดครั้งเดียวแล้วปล่อยให้ตัวเลขสีแดงพูดเกินจริงเอง

## 2. หลักปรัชญาการออกแบบ (Design Principles)

1. **Density with clarity** — ข้อมูลแน่นแบบ financial/analytics dashboard (คิดถึง Bloomberg Terminal, Linear, Stripe Dashboard) แต่ทุกกลุ่มข้อมูลต้องมี label ชัดเจน ไม่ให้ผู้ใช้ต้องเดา
2. **Progressive disclosure** — ซ่อนความซับซ้อนไว้เบื้องหลัง แสดงเมื่อจำเป็น: กล่องแหล่งข้อมูลย่ออัตโนมัติหลังโหลดเสร็จ, พารามิเตอร์ขั้นสูงอยู่ใน `<details>` ที่พับได้, คำอธิบายศัพท์เทคนิคซ่อนใน tooltip ไม่ใช่ข้อความลอย
3. **Restraint in color** — สีมีความหมายตายตัว ห้ามใช้พร่ำเพรื่อ (ดูระบบสีข้อ 3.1) แดง = สัญญาณเสี่ยง/เตือนเท่านั้น ไม่ใช้แดงเพื่อความสวยงาม
4. **Every number earns its place** — ไม่มีตัวเลขลอยที่ไม่มี label/unit กำกับ ตัวเลขทุกตัวใช้ font ตัวเลข monospace เพื่อให้อ่านแนวตั้งง่าย (ราคา, เปอร์เซ็นต์)
5. **Trust through transparency** — ทุกหน้าจอบอกที่มาข้อมูลและข้อจำกัด ไม่ปิดบังว่าเป็นการประมวลผลอัตโนมัติ
6. **Zero server, zero friction** — ไม่มีการ login, ไม่มี loading spinner เปล่าๆ ไม่มี dead-end (ทุก error state มีทางออกเสมอ เช่น "เลือกไฟล์จากเครื่องแทน")

## 3. Design System

### 3.1 ระบบสี (Color Tokens)

```css
--ink:      #0E1B33   /* ข้อความหลัก, หัวเรื่อง, พื้นหลัง header */
--paper:    #EDF0F6   /* พื้นหลังหน้าเว็บ (เทาอมฟ้าอ่อนมาก) */
--surface:  #FFFFFF   /* พื้นหลังการ์ด/คอมโพเนนต์ */

--blue:     #1D4E89   /* Primary — ปุ่มหลัก, active state, ตัวเลขสำคัญ */
--blue2:    #3A78C2   /* Secondary blue — ลิงก์, accent รอง */
--blueSoft: #E6EEF8   /* พื้นหลังอ่อน — badge, info box, ปุ่ม active-soft */

--red:      #C4341C   /* Alert — ธงความเสี่ยง, ตัวเลขเตือน, ป้ายผิดปกติ */
--redSoft:  #FBE9E5

--amber:    #D99400   /* Warning — สถานะกึ่งเสี่ยง, hbar สีเตือน */
--amberSoft:#FBF1DC

--green:    #1B7F5A   /* Positive — สถานะแคชแล้ว, success */
--greenSoft:#E4F1EB

--muted:    #5C6672   /* ข้อความรอง, label */
--muted2:   #8A93A3   /* ข้อความระดับสาม, placeholder, ค่าที่ไม่มี */

--line:     #D6DCE8   /* เส้นขอบการ์ด/ตาราง */
--line2:    #E9EDF3   /* เส้นแบ่งบรรทัดในตาราง (เบากว่า) */

--shadow: 0 1px 2px rgba(14,27,51,.05), 0 8px 22px rgba(14,27,51,.05);
```

**กฎการใช้สี:**
- แดงใช้เฉพาะกับสัญญาณความเสี่ยง/คำเตือนข้อมูลผิดปกติเท่านั้น ห้ามใช้ตกแต่ง
- เขียวสงวนไว้สำหรับ "สถานะสำเร็จของระบบ" (แคชแล้ว) ไม่ใช่ "ผลลัพธ์ดี" ของการวิเคราะห์ — เพราะการวิเคราะห์นี้ไม่มี "ดี/ไม่ดี" มีแต่ "เข้าเกณฑ์/ไม่เข้าเกณฑ์"
- น้ำเงินเข้ม (`--ink`) ใช้เป็นพื้นหลัง header แบบ gradient: `linear-gradient(135deg, #0E1B33, #1B3A66)` เพื่อให้ดู authoritative แบบเอกสารราชการ/สถาบันการเงิน ไม่ใช่ playful

### 3.2 ตัวอักษร (Typography)

| การใช้งาน | Font | Weight | หมายเหตุ |
|---|---|---|---|
| เนื้อหาทั่วไป, UI, ภาษาไทย | **Sarabun** | 300/400/500/600/700/800 | โหลดจาก Google Fonts, รองรับวรรณยุกต์ไทยครบ |
| ตัวเลข, ราคา, สถิติ, รหัสโครงการ | **IBM Plex Mono** | 500/600 | `font-variant-numeric: tabular-nums` เพื่อให้ตัวเลขจัดคอลัมน์ตรงกัน |

- Base font-size: `15px` (body) — เหมาะกับ dashboard ที่ต้องแสดงข้อมูลจำนวนมากในจอเดียว
- หัวข้อ h3 ในการ์ด: `15px, weight 800`
- Caption/label: `10.5–12.5px, weight 600–700, letter-spacing .03–.06em, uppercase` สำหรับ label ระดับ micro (หัวตาราง, kpi label)
- ตัวเลขเด่นใน KPI card: `19–26px, weight 800`

**กฎ:** ทุกที่ที่แสดงตัวเลข (บาท, เปอร์เซ็นต์, จำนวนสัญญา) ต้องห่อด้วย class ที่กำหนด `font-family: IBM Plex Mono` — ห้ามปล่อยให้ Sarabun render ตัวเลข เพราะตัวเลขใน Sarabun ไม่ align แนวตั้งสวย

### 3.3 Spacing, Radius, Elevation

- Border-radius แบ่งตามลำดับชั้นความสำคัญของ component:
  - ปุ่มเล็ก/badge: `6–8px`
  - ปุ่มมาตรฐาน, input: `9–10px`
  - การ์ด/พาแนล: `11–13px`
  - Modal: `16px`
  - Chip/pill (ตัวกรอง): `16–18px` (โค้งมนเต็ม)
- เงา (shadow) ใช้แบบ layered เบามาก 2 ชั้นซ้อนกัน (`--shadow`) ให้ความรู้สึก "ลอยเบาๆ" ไม่ dramatic แบบ material design ทั่วไป
- Card padding มาตรฐาน: `17px`
- Gap ระหว่าง grid items: `12–15px`

### 3.4 Component Library

**Buttons** (4 variants, ทุกตัว `border-radius:10px, font-weight:700`)
- `.btn` — พื้นน้ำเงิน ตัวหนังสือขาว (primary action: "โหลดและวิเคราะห์")
- `.btn.sec` — พื้นขาว ขอบ/ตัวหนังสือน้ำเงิน (secondary: "Export CSV")
- `.btn.ghost` — พื้นขาว ขอบเทา ตัวหนังสือเทา (tertiary: "ค้นหาใหม่", ปุ่มเสริมในตาราง)
- `.btn.warn` — พื้นแดงอ่อน ขอบ/ตัวหนังสือแดง (destructive: ล้างข้อมูล)
- ทุกปุ่มมี size variant `.sm` (padding เล็กลง, font 12–13.5px) สำหรับใช้ในหัวการ์ด/แถวตาราง

**KPI Card** (`.kpi`)
- การ์ดขาว มีแถบสีบาง 4px ที่ขอบซ้ายบอกสถานะ (ปกติ = น้ำเงิน, alert = แดง, ok = เขียว)
- โครงสร้าง: label เล็ก (เทา, uppercase) → ตัวเลขใหญ่ (monospace, หนา) → unit เล็ก (เทาอ่อน)
- จัด grid 6 คอลัมน์บน desktop, ยุบเหลือ 3 แล้ว 2 บนจอเล็ก

**Flag Card** (`.flag`) — การ์ดแสดง "ธงความเสี่ยง"
- ต่างจาก KPI card ตรงมีแถบสีซ้าย **หนา 5px สีแดง** เป็นค่าเริ่มต้น (เพราะทุกใบคือสัญญาณเสี่ยง) และ **เปลี่ยนเป็นเทาเมื่อค่า = 0** (`.off` state) เพื่อลดความสำคัญของสิ่งที่ไม่พบ
- คลิกได้ (ใช้เป็นตัวกรองตาราง) — เมื่อเลือกจะมี `outline` สีแดงหนา 2.5px ล้อมรอบ (`.sel` state)
- มี info icon (ⓘ) มุมชื่อการ์ด เปิด tooltip อธิบายเกณฑ์แบบเต็มโดยไม่ต้องออกจากหน้า

**Chips** (`.chip`) — ตัวกรองแบบ multi-select (จังหวัด/ปี)
- Pill shape, ขอบเทาบาง 1.5px, พื้นขาว
- Active state: พื้นเข้ม (`--ink` สำหรับจังหวัด, `--blue` สำหรับปี — แยกสีตามมิติข้อมูลเพื่อไม่ให้สับสน) ตัวหนังสือขาว
- รองรับ indeterminate state (checkbox บางส่วนถูกเลือก) แสดงเป็นขีดกลาง (—) แทนติ๊ก

**Tabs** (`.vtab`) — สลับมุมมองการวิเคราะห์
- แถวปุ่มแนวนอน มี icon (emoji) นำหน้าชื่อมุมมอง เพื่อให้ scan ด้วยตาเร็ว
- Active tab: พื้นเข้ม (`--ink`), inactive: ขอบเทาบาง

**Table**
- Header sticky (`position:sticky; top:0`) เมื่อ scroll ในกล่อง
- Header คลิกได้ทุกคอลัมน์เพื่อเรียง — ระบบ auto-detect ชนิดข้อมูล (ตัวเลข / วันที่ไทย / ข้อความ) แล้วเรียงให้ถูกวิธี พร้อมลูกศร ▲▼ บอกทิศทาง
- คอลัมน์ตัวเลขชิดขวา (`.num`), จัด font monospace
- แถวมี hover state อ่อนๆ (`background:#F8FAFD`)
- ทุกตารางยาวอยู่ในกล่อง `.scroll` ที่จำกัดความสูง ป้องกันตารางยาวดันหน้าเว็บ

**Tooltip** (`.info` + `.tip`)
- ไอคอนวงกลมเล็ก "i" ข้าง label — โฟกัส/hover แล้วมี bubble ข้อความลอยขึ้น (ไม่ใช่ native title attribute เพราะต้องคุมสไตล์และรองรับ multi-line ภาษาไทย)
- ใช้เพื่อซ่อนคำอธิบายศัพท์เทคนิค/เกณฑ์คำนวณ โดยไม่ทำให้หน้าจอรก — ทุกพารามิเตอร์และธงมี tooltip แบบนี้

**Modal** (`.mask` + `.modal`)
- Backdrop มืด 60% opacity, modal ขาวลอยกลางจอ border-radius 16px
- ใช้สำหรับ "วิธีใช้งาน" ไม่ใช่สำหรับ critical action — เปิด/ปิดได้ 3 ทาง (ปุ่ม ✕ / คลิก backdrop / กด Esc) เพื่อลด friction
- เนื้อหาภายในใช้ pattern "numbered steps" (วงกลมเลข + หัวข้อหนา + คำอธิบาย) ไม่ใช่ paragraph ยาว

**Progress bar** (`.prog`)
- แถบบางแสดงระหว่างโหลด/ประมวลผลไฟล์ใหญ่ พร้อมข้อความสถานะละเอียด (ชื่อไฟล์ที่กำลังอ่าน, จำนวนแถวที่ประมวลผลแล้ว) ไม่ใช่แค่ spinner เฉยๆ — สำคัญมากเพราะไฟล์อาจใหญ่ 100MB+ ผู้ใช้ต้องรู้ว่าระบบยังทำงานอยู่ไม่ค้าง

**Badge**
- Pill เล็กมาก มีจุดสี ใช้บอกสถานะแคช (เขียว = มีแคชแล้ว) ในตารางเลือกชุดข้อมูล

**HTML Bar chart** (custom, ไม่ใช้ library)
- แถบแนวนอนแบบ minimal: label ซ้าย → ค่า/เปอร์เซ็นต์ขวา → แถบสี fill ด้านล่าง
- ใช้แทนกราฟแท่งเต็มรูปแบบในจุดที่ไม่คุ้มโหลด library เพิ่ม (เช่น สัดส่วนวิธีจัดหา, หน่วยงานที่พบกลุ่มเสี่ยงมากสุด)

## 5. รายละเอียดแต่ละมุมมอง (View-specific UX)

ทุกมุมมองใช้โครงเดียวกัน: **KPI แถวบน → เนื้อหาหลัก (2/3 ความกว้าง) + side panel (1/3)** เพื่อให้ผู้ใช้คุ้นเคยเมื่อสลับมุมมอง

**🕸 เครือข่ายความสัมพันธ์**
- ช่องค้นหาอยู่เหนือกราฟโดยตรง (ไม่ใช่ในแถบตัวกรองบนสุด) เพราะเป็นการกระทำที่ผูกกับกราฟเท่านั้น
- กราฟ (vis-network): จุดสีน้ำเงินอ่อน = หน่วยงาน, จุดสีส้ม/แดง = ผู้รับจ้าง (แดงถ้าติดธง), ขนาดจุด ∝ มูลค่า, ความหนาเส้น ∝ มูลค่าความสัมพันธ์
- การ์ดธง 6 ใบอยู่ *ใต้* กราฟและ *เหนือ* ตารางผู้รับจ้าง — ตำแหน่งนี้สื่อว่าการ์ดธงเป็นตัวกรองของตารางด้านล่าง ไม่ใช่แค่สรุปตัวเลข
- Side panel: กล่อง "การกระจุกมูลค่า" (Top 10/20/50/100 %) + กล่อง "รายละเอียด" ที่อัปเดตเนื้อหาเมื่อคลิกจุดในกราฟหรือแถวในตาราง (ไม่เปิด modal ใหม่ — คงบริบทให้ผู้ใช้)

**📑 แบ่งซื้อแบ่งจ้าง / 🎯 วิธีเฉพาะเจาะจง / 💰 ราคาและส่วนลด**
- แต่ละมุมมองมี KPI แถวบนเฉพาะของตัวเอง (4–5 ใบ) ตามด้วยตารางหลัก + side panel แสดง bar chart สรุป
- ตารางที่ผิดปกติมากมี "หมายเหตุ" ต่อท้ายแถว (เช่น ป้าย "⚠ ข้อมูลอาจคลาดเคลื่อน — ตรวจสอบใน e-GP" เมื่อราคาสูงผิดปกติ) — ใช้ tag สีแดงเล็กในเซลล์สุดท้าย ไม่ทำทั้งแถวเป็นสีแดง (จะรบกวนการอ่านตัวเลขอื่นในแถว)

**Drill-down pattern (ใช้ร่วมทุกมุมมอง)**
- ทุกที่ที่มีชื่อผู้รับจ้าง/หน่วยงาน มีปุ่มเล็ก "รายละเอียด" ต่อท้าย
- คลิกแล้ว: เปิดตารางใหม่ท้ายหน้า, auto-scroll ไปหา, มีปุ่ม "↑ กลับไป [จุดที่มาก่อนหน้า]" ระบุปลายทางกลับตามบริบท (กลับกราฟ / กลับตาราง trend / กลับมุมมองเฉพาะเจาะจง) — ไม่ใช้ "กลับ" คำเดียวลอยๆ
- ชื่อโครงการในตาราง detail เป็นลิงก์คลิกเปิดแท็บใหม่ไปหน้าค้นหา e-GP โดยตรง

## 6. Interaction & Micro-behavior Rules

- **Auto-collapse:** ส่วนที่ทำหน้าที่เสร็จแล้ว (กล่องเลือกไฟล์) ย่อตัวเองทันทีหลังโหลดสำเร็จ ลด scroll โดยไม่ต้องให้ผู้ใช้สั่ง
- **Session persistence:** ข้อมูลที่โหลดแล้วบันทึกอัตโนมัติ (IndexedDB) — รีเฟรชหน้าไม่ต้องโหลดใหม่ พร้อมแถบแจ้งเตือนสีเขียวอ่อนบอกว่ากู้คืนข้อมูลเมื่อไร
- **Non-blocking heavy compute:** ไฟล์ใหญ่ประมวลผลผ่าน Web Worker + แบ่งเป็น chunk คืนการควบคุมให้ UI เสมอ — ห้าม freeze หน้าจอแม้ไฟล์เป็นร้อย MB
- **Graceful degradation ของการค้นหาแหล่งข้อมูล:** ลองหลายวิธี (GitHub API → manifest.json → directory listing → แคชเดิม → เลือกไฟล์เอง) และ **แจ้งเตือนเมื่อบางส่วนหาไม่เจอ** แทนที่จะเงียบหาย
- **Sort ไม่ทำลาย state อื่น:** คลิกหัวตารางเรียงลำดับ ไม่รีเซ็ต filter ที่เลือกไว้
- **ปุ่มปิด popup 3 ทางเสมอ**
- **ทุก empty state มีข้อความอธิบาย** ("ไม่พบรายการในขอบเขตที่เลือก") ไม่ปล่อยตารางว่างเปล่าให้เดา

## 7. Responsive Behavior

- Breakpoint หลัก: `1000px` และ `600px`
- `>1000px`: KPI grid 6 คอลัมน์, flag grid 6 คอลัมน์, side panel คงที่ 295–340px
- `≤1000px`: KPI/flag grid ยุบเหลือ 3 คอลัมน์, กราฟ+side panel ซ้อนเป็นคอลัมน์เดียว, parameter panel ยุบเหลือ 2 คอลัมน์
- `≤600px`: KPI/flag grid ยุบเหลือ 2 คอลัมน์
- Design นี้เป็น **desktop-first โดยเจตนา** (เครื่องมือนักวิเคราะห์ใช้จอใหญ่เป็นหลัก) แต่ยังใช้งานได้บนแท็บเล็ต/มือถือแบบไม่พัง

## 8. Content & Microcopy Guidelines

- ภาษาไทยทางการตลอดทั้งระบบ (ไม่ใช้ภาษาพูด/แสลง)
- ป้ายกำกับปุ่มขึ้นต้นด้วย verb ชัดเจน + emoji ไอคอนนำ (⬇ ดาวน์โหลด, ⟳ ค้นหาใหม่, 🗑 ล้าง)
- Disclaimer ใช้โทน "ข้อควรระวัง" ไม่ใช่ "คำเตือนทางกฎหมาย" — เน้นให้ผู้ใช้ตีความอย่างระมัดระวัง ไม่ใช่ปัดความรับผิดชอบ
- ทุก tooltip เขียนแบบ "ระบบจะ... เมื่อ..." (บอกเงื่อนไข mechanic ตรงไปตรงมา) ไม่ใช้ภาษาการตลาด

## 9. สรุปเป็น Prompt เดียว (ใช้วางในเครื่องมือ AI generation ได้ทันที)

> ออกแบบ single-page analytics dashboard สำหรับนักตรวจสอบภาครัฐ โทนสี navy-blue (#0E1B33, #1D4E89) บนพื้นเทาอมฟ้าอ่อน (#EDF0F6) การ์ดขาวขอบบางเงาเบา ใช้ฟอนต์ Sarabun สำหรับข้อความและ IBM Plex Mono (tabular-nums) สำหรับตัวเลขทั้งหมด สไตล์ให้ความรู้สึก "เอกสารราชการที่ทันสมัย" ไม่ playful ไม่มี gradient สีสันฉูดฉาด ยกเว้น header ที่เป็น dark gradient เดียว
>
> โครงสร้าง: header เข้ม → การ์ดเลือกข้อมูลที่ย่อตัวเองอัตโนมัติหลังใช้งาน → แถว KPI card 6 ใบ (แถบสีซ้าย 4px บอกสถานะ) → filter chips → parameter panel พับได้ → menu tabs สลับมุมมอง 4 แบบ → เนื้อหาหลัก layout 2:1 (เนื้อหา + side panel) → disclaimer box สีเหลืองอ่อนท้ายเสมอ
>
> ทุก metric ที่เป็น "สัญญาณเสี่ยง" ใช้การ์ดขอบซ้ายสีแดงหนา 5px คลิกได้เพื่อกรองตารางด้านล่าง เปลี่ยนเป็นเทาเมื่อค่าเป็นศูนย์ ทุก parameter/เกณฑ์มี info-icon เปิด tooltip อธิบายละเอียดแทนการเขียนคำอธิบายลอยในหน้า ตารางทุกตัวคลิกหัวคอลัมน์เพื่อเรียงได้ (auto-detect ตัวเลข/วันที่/ข้อความ) แถวมีปุ่ม "รายละเอียด" เปิดตาราง drill-down ท้ายหน้าพร้อม auto-scroll และปุ่มย้อนกลับที่รู้ปลายทาง
>
> Interaction ต้อง non-blocking เสมอแม้ประมวลผลข้อมูลหนัก, มี progress indicator ละเอียด, บันทึก state อัตโนมัติเพื่อกู้คืนหลัง refresh, ทุก empty/error state มีคำอธิบายและทางออกเสมอ ไม่มี dead-end
